# issues.md 编写规范

## 创建时机

发现 Bug 或技术债时追加一条记录。文件在首次发现问题时创建。

## 文件位置

`docs/<branch-id>/issues.md`

## 模板

```markdown
# 问题追踪

> 分支：`<branch-name>`

---

## 问题 #N

**发现时间：** YYYY-MM-DD HH:MM

**类型：** Bug / 技术债 / 性能问题 / 安全风险

**严重程度：** 阻塞 / 严重 / 一般 / 轻微

**描述：**
<!-- 清晰描述问题现象 -->

**复现步骤（Bug 类）：**
1. <!-- 步骤 1 -->
2. <!-- 步骤 2 -->

**影响范围：**
<!-- 受影响的功能/用户/数据 -->

**是否本分支修复：** 是 / 否

**修复结果（如已修复）：**
<!-- 修复的 commit hash 和简要说明 -->

**遗留标记（如不修复）：**
<!-- 不修复的原因，以及计划在何时处理 -->
```

## 内容要求

### 类型

- **Bug**：代码缺陷导致的行为异常
- **技术债**：当前能工作但存在长期隐患的代码/设计
- **性能问题**：可测量的性能不达标
- **安全风险**：潜在或被证实的漏洞

### 严重程度

- **阻塞**：不修复则功能不可用或数据有损
- **严重**：影响核心功能或用户体验
- **一般**：影响非核心功能或有 workaround
- **轻微**：不影响功能，仅美观或规范问题

### 描述

- 说现象，不说原因（原因留到修复时分析）
- 加上期望行为 vs 实际行为的对比

### 复现步骤

- 必须是他人可按步骤复现的
- 包含环境信息（浏览器版本、操作系统等）
- 如果无法稳定复现，标注"偶现"并描述触发条件

### 是否本分支修复

- **是**：修复后在此条追加修复结果
- **否**：标注遗留原因和计划处理时间

### 追加规则

- 每个问题独立编号（从 1 开始递增）
- 已修复的问题不要删除，追加修复结果即可
- 问题之间用 `---` 分隔
