# 0-1 初建架构设计工作流

适用于：从空项目或新子系统开始建立**初始技术架构**。

入口：由 `SKILL.md` 路由进入。

---

## 输入要求

开始前必须从用户或既有材料中获得以下信息。任何一项缺失，必须先追问，不得自行假设。

| 输入项 | 内容 | 缺失时的处理 |
|---|---|---|
| **产品目标** | 系统要解决什么问题，给谁用，价值在哪里 | 追问，禁止猜测 |
| **核心能力** | 第一阶段必须支持哪些能力 | 追问 |
| **非目标** | 明确不做什么，避免范围蔓延 | 追问 |
| **核心实体** | 系统中有哪些主要业务对象、它们的关系 | 与目标一起追问 |
| **质量要求** | 性能 / 安全 / 可用性 / 可维护性 / 可观测性 | 追问，但允许给出"无特殊要求"的默认值 |
| **技术约束** | 语言、框架、数据库、部署环境、团队习惯、合规要求 | 必须明确，给出默认值前先说明 |
| **验证方式** | 怎么判断第一版架构是可用的 | 追问 |

---

## 工作流步骤

按顺序执行。任何一步未完成不得跳到下一步。

### 步骤 1：识别系统类型

判断主要类型，确定关注点：

| 类型 | 关注点 |
|---|---|
| 前端应用 | 路由、状态、组件边界、数据获取、构建产物 |
| 后端 API | 分层、Repository、事务、错误模型、鉴权 |
| 全栈应用 | 前后端契约、跨层数据流、部署形态 |
| CLI 工具 | 命令路由、I/O、配置、退出码 |
| 数据处理任务 | Pipeline、可重试、幂等、监控 |
| 桌面 / 移动应用 | 平台 API、离线 / 同步、生命周期 |
| 多服务系统 | 服务边界、通信方式、数据所有权 |
| SDK / 插件 | 公开 API、版本、向后兼容 |

不同类型的关注点不同，禁止用一套模板套所有系统。

### 步骤 2：提炼核心业务模型

输出必须包括：

```markdown
## 核心模型

- 实体 A：作用、关键字段
- 实体 B：作用、关键字段
- 实体之间的关系（1:1 / 1:N / M:N）

## 状态与生命周期

- 实体 X 的状态：S1 / S2 / S3
- 允许的状态转移：S1 → S2、S2 → S3

## 关键业务规则

- 规则 1（明文说明）
- 规则 2

## 不变量

- 不变量 1（始终成立）
- 不变量 2
```

业务规则与不变量是 Domain 层的设计依据，禁止在此阶段省略。

### 步骤 3：提出候选架构方案

至少给出 2 个、最多 3 个方案，每个方案包括：

- 架构风格（分层单体 / 模块化单体 / 微服务 / 事件驱动 / CQRS 等）
- 主要分层
- 主要模块
- 主要依赖与外部组件
- 优点、缺点、风险
- 适合的规模与团队

然后给出**推荐方案**并明确取舍理由。

**禁止**：

- 跳过候选直接给一个方案。
- 把"未来可能演进到微服务"作为现在就选微服务的理由。

### 步骤 4：定义模块边界

输出格式：

```markdown
## 模块划分

| 模块 | 职责 | 不负责 | 对外契约 | 内部允许变更 |
|---|---|---|---|---|
| user | 用户注册、登录、资料 | 权限策略细节 | UserService | 内部 Repository、缓存策略 |
| project | 项目管理 | 任务执行 | ProjectService | 内部存储与索引 |
| task | 任务状态与规则 | 用户认证 | TaskService | 内部状态机实现 |

## 模块通信方式

- 模块间通过 Service 接口调用，不直接访问对方 Repository。
- 跨模块的复杂流程通过 Application 层编排。
- 异步事件（如有）通过事件总线，事件契约集中定义。
```

### 步骤 5：定义分层与依赖规则

输出格式：

```markdown
## 分层

- Interface 层：HTTP / CLI / 消息消费者入口
- Application 层：用例编排、事务边界、权限入口
- Domain 层：实体、值对象、领域服务、业务规则
- Infrastructure 层：DB、缓存、MQ、文件、外部 API

## 依赖规则

- Interface → Application
- Application → Domain 接口
- Infrastructure → 实现 Domain / Application 定义的接口
- Domain 不依赖 Interface / Infrastructure / 框架
- 禁止循环依赖
- 禁止业务模块互相直接依赖对方内部实现
```

### 步骤 6：定义数据访问模式

必须明确：

```markdown
## 数据访问

- 数据访问入口：Repository 接口
- ORM / Driver：[名称、版本]
- 事务边界：由 Application 层 UseCase 控制，不在 Repository 内开事务
- 查询模型与领域模型：分离 / 共用（说明理由）
- 跨模块数据访问：禁止，必须通过对方 Service 接口
- 数据库错误：在 Repository 实现中转换为业务错误类型
```

### 步骤 7：定义接口契约

API、函数、事件、错误的契约必须有初始草案：

```markdown
## API 草案

| 方法 | 路径 | 输入 | 输出 | 错误 |
|---|---|---|---|---|
| POST | /users | RegisterUserDTO | UserDTO | 400 / 409 |
| POST | /sessions | LoginDTO | SessionDTO | 400 / 401 |

## 错误模型

- DomainError：业务规则违反
- AuthError：认证 / 授权失败
- ValidationError：参数校验失败
- InfrastructureError：外部依赖错误
- SystemError：未分类错误

## 错误映射

- DomainError → 400 / 409（按子类决定）
- AuthError → 401 / 403
- ValidationError → 400
- InfrastructureError → 502 / 503
- SystemError → 500
```

### 步骤 8：定义配置与环境

```markdown
## 配置

- 来源：环境变量 + 配置文件
- 集中读取位置：[模块名]
- 集中校验位置：启动时
- Secret 来源：[来源]，禁止进入代码、日志、测试快照
- 多环境差异：通过环境变量切换，不通过代码分支
```

### 步骤 9：定义测试与验证策略

```markdown
## 测试策略

- 单元测试：覆盖 Domain 规则、纯函数、关键 Application UseCase
- 集成测试：覆盖 Repository、外部适配器
- E2E 测试：覆盖关键用户路径（注册、登录、核心业务流）
- mock 策略：Domain 单元测试不 mock；Application 单元测试 mock Repository；Infrastructure 集成测试使用测试容器或测试数据库
- 架构边界检查：[工具或人工 review 流程]
```

### 步骤 10：写入设计文档

按 `reference.md` 中的「架构设计文档模板」写入 `docs/<branch>/design.md` 的「架构设计」节。

涉及全局文档（ARCHITECTURE / API / DATABASE）的部分**不在初建阶段写入**，由 `/accept` 合入流程统一同步。

## 0-1 检查清单

写入设计文档之前自检：

- [ ] 产品目标已明确
- [ ] 非目标已明确
- [ ] 核心业务模型已写出（实体、关系、状态、规则、不变量）
- [ ] 已识别系统类型
- [ ] 至少 2 个候选架构方案
- [ ] 推荐方案与取舍理由已写明
- [ ] 模块边界与职责已定义
- [ ] 分层与依赖规则已定义
- [ ] 数据访问模式已定义
- [ ] 接口契约草案已写出
- [ ] 错误模型已定义
- [ ] 配置与 Secret 管理方式已定义
- [ ] 测试策略已定义

- [ ] 验收标准已写明

任何一项未通过，禁止进入实现。

## 反模式

| 反模式 | 应做法 |
|---|---|
| 上来就画完整微服务图 | 先判定规模，默认单体 / 模块化单体 |
| 把所有"可能要变的地方"都抽象出来 | 只为当前需求设计；变化点等出现第二个用例再抽象 |
| 设计目录结构作为架构 | 先定边界、依赖、契约，目录服务于这些 |
| 把数据库 schema 当作领域模型 | Domain 模型独立设计，Repository 负责映射 |
| 在 Controller 写业务规则 | 业务规则属于 Domain；Controller 只做入参出参与错误转换 |
| 复制三个 Service 之间相同的"权限检查" | 抽取领域服务或权限模块 |
| 用"未来可能换数据库"作为引入抽象的理由 | 除非真有可见的切换计划，否则不抽象 |
