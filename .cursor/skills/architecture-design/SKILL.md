# Architecture Design

为 AI 提供严谨、可执行的软件架构设计指导。本技能不替代具体业务设计，只规定**做架构设计时必须遵循的流程、产物和边界**。

适用两种场景：


| 场景         | 触发                                          | 详细工作流                    |
| ---------- | ------------------------------------------- | ------------------------ |
| **0-1 初建** | 新建产品、服务、应用、CLI、SDK 等                        | `workflow-greenfield.md` |
| **1-N 迭代** | 新增功能 / 修缺陷 / 重构 / 性能 / 安全 / 数据模型变更 / 外部依赖变更 | `workflow-iteration.md`  |


参考资料：

- 架构核心要素、文档模板、检查清单 → `reference.md`

---

## 触发条件

只要任务触及以下任一项，本技能必须被加载：

- 新建系统或新建子系统
- 新增、删除或重命名模块
- 调整层与层之间的依赖
- 新增 / 修改公开 API 或事件契约
- 改变核心数据模型或表结构
- 引入新的外部依赖（DB、MQ、第三方 SDK）
- 跨模块的代码迁移或重构
- 关于"是否要重构 / 是否要拆服务 / 是否要分层"的提问
- 同一类业务规则被复制到多处，开始讨论收敛

**禁止**：在未识别上述任一情况之前直接编码或直接给出"重构方案"。

## 基本原则（必须遵守）

### 1. 目标先行

在任何架构方案前必须明确：

- **目标**：本次架构设计要解决什么。
- **非目标**：本次明确不解决什么。
- **不变量**：哪些已有行为不能被破坏。
- **约束**：语言、框架、数据库、部署、团队习惯、合规要求。

四项缺一不可。缺失项必须先追问，不得自行假设。

### 2. 边界先于目录

架构设计的核心是边界，不是文件夹结构。

必须先回答：

- 系统有哪些模块？
- 每个模块做什么、不做什么？
- 模块间如何通信？
- 哪些依赖方向允许、哪些禁止？

目录结构服务于这些边界，而不是替代它们。

### 3. 依赖方向单向

推荐方向：

```text
Interface / API / UI
    → Application / UseCase
        → Domain
            ← Infrastructure 实现 Domain 所需接口
```

硬性约束：

- 业务核心（Domain）不得依赖 Web 框架、ORM、HTTP、环境变量。
- Infrastructure 不得编排业务流程。
- 禁止循环依赖。
- 禁止业务模块互相直接访问对方内部实现，只共享契约。

### 4. 业务规则集中

业务规则必须集中在领域层或明确的业务服务中，禁止散落到：

- Controller / API handler
- SQL / ORM hook
- 前端按钮逻辑
- 临时脚本
- 第三方服务适配器

### 5. 不为未来过度设计

只为当前明确的需求设计。

- 未来可能但未发生的需求 → 不引入抽象。
- 真实变化点（已知会变的支付、登录、存储、模型供应商）→ 用接口隔离。
- "为了优雅"的抽象 → 拒绝。

### 6. 架构必须可验证

每项架构决策都应有对应验证手段：

- 目录结构体现模块边界。
- 依赖方向可通过静态检查或代码审查验证。
- 核心业务规则有单元测试。
- 跨模块流程有集成测试。
- 关键用户路径有 E2E 测试。

## AI 行为约束

### 必须

1. **先判断场景**：本次属于 0-1 还是 1-N。
2. **先明确目标 / 非目标 / 不变量 / 约束**。
3. **1-N 必须先调查现状**：相关模块、调用链、数据流、测试覆盖、兼容性约束。
4. **至少给出两个候选方案**，明确取舍与推荐。
5. **明确模块边界、依赖方向、数据访问、错误处理、测试策略**。
6. **架构调整必须给出迁移步骤与回滚策略**。
7. **写入设计文档的「架构设计」节**（位置与分支文档结构由项目文档规则决定，本技能不在此处复述）。

### 禁止

1. 不了解现状就提重构。
2. 不明确目标就设计目录结构。
3. 只给一个方案而不说取舍。
4. 引入无需求驱动的抽象。
5. 把未来可能需求当作当前必需需求。
6. 在 1-N 场景默认推翻现有架构。
7. 把业务规则散落到入口层或基础设施层。
8. 跳过测试策略 / 跳过兼容性分析 / 跳过迁移与回滚。
9. 仅输出"架构图"而不输出可执行步骤。

## 执行流程（总览）

```text
[场景识别]
     │
     ├── 0-1 初建 ── 读 workflow-greenfield.md 按步骤执行
     │
     └── 1-N 迭代 ── 读 workflow-iteration.md
                       │
                       ├── 现状调查
                       ├── 变化点定位
                       ├── 调整判断：不调整 / 局部调整 / 系统级调整
                       ├── 候选方案 + 推荐
                       ├── 迁移策略 + 回滚策略
                       └── 写入 design.md
```

无论哪个场景，最终都需输出一份符合 `reference.md` 模板的「架构设计」章节内容（写入位置由项目文档规则决定）。

## 何时升级、何时收敛


| 现象             | 处理                                |
| -------------- | --------------------------------- |
| 用户描述含多个独立子系统   | 先拆分为子项目，每个子项目各自走本技能 |
| 现有系统只是局部加一个功能  | **优先**走 1-N 不调整路径，不要顺手重构          |
| 反复出现相同的"散落规则"  | 1-N 局部调整：抽取领域服务                   |
| 数据模型不支持新不变量    | 1-N 系统级调整：先改模型再加功能                |
| 多个模块边界已变模糊     | 1-N 系统级调整：先划边界再加功能                |
| 设计中出现"为了未来扩展性" | 立即收敛：删除该抽象，直到出现第二个真实用例            |

## 产物

每次完成架构设计后，必须输出以下内容（具体写入位置与文件名由项目文档规则决定，本技能不复述）：

1. 「架构设计」章节内容（必填）。
2. 数据库变化的增量描述（若涉及）。
3. API 变化的增量描述（若涉及）。
4. 全局架构变化的增量描述（若涉及）。
5. 带兼容风险时的风险记录与回滚策略。

文档模板与字段要求见 `reference.md`。
