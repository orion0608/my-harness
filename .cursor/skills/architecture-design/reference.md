# 架构设计参考

本文档提供：

1. 架构核心要素的定义与默认约束
2. 架构设计文档模板（写入 `design.md` 的「架构设计」节）
3. AI 推荐提示词模板

入口由 `SKILL.md` 路由。本文按需查阅，不必通读。

## 一、架构核心要素

### 1.1 分层

常用分层与职责：

| 层 | 职责 | 不负责 |
|---|---|---|
| Interface / API / UI | HTTP / CLI / 消息消费者 / 页面入口、参数解析、鉴权入口 | 业务规则、事务、数据库操作 |
| Application / UseCase | 用例编排、事务边界、权限入口、调用领域服务与外部接口 | 表达业务规则细节、直接操作数据库 |
| Domain | 实体、值对象、领域服务、业务规则、状态机、不变量 | HTTP / DB / 框架细节、外部 SDK |
| Infrastructure | DB、缓存、MQ、文件、外部 API 实现 | 业务规则编排 |

设计时必须显式说明：

- 是否采用分层架构
- 每一层职责与禁区
- 允许的依赖关系
- 任何例外的理由

### 1.2 模块

两种主要划分方式：

| 方式 | 适用 | 优点 | 风险 |
|---|---|---|---|
| 按技术分层 (controllers / services / repositories / models) | 小型项目 | 简单、好上手 | 业务大了散落严重 |
| 按业务能力分模块（每模块内部各自分层） | 中大型项目 | 模块边界清晰、利于 AI 协作 | 初期开销略高 |

每个模块的描述必须包括：

- 职责
- 不负责什么
- 对外公开契约（接口 / DTO / 事件）
- 内部允许变更的部分

### 1.3 依赖治理

硬性约束：

- 依赖方向单向，业务核心不依赖框架与基础设施。
- 业务模块之间**只共享契约**，不共享内部实现。
- 禁止循环依赖。
- 公共工具（logger / config / db / auth）单独维护，业务模块只读使用。
- 跨模块通信优先使用 Service 接口；异步场景使用事件，事件契约集中定义。

### 1.4 数据访问

必须明确：

- 谁可以访问数据库（默认仅 Repository）
- 使用 Repository / DAO / ORM Model / Query Service
- 事务边界（默认在 Application 层 UseCase 开启）
- 查询模型与领域模型是否分离
- 数据库错误如何转换为业务错误
- 是否允许跨模块 join 表（默认禁止）
- 是否允许 API 层直接使用 ORM 对象（默认禁止）

### 1.5 接口契约

接口包含：API、函数、事件、错误类型、数据模型、外部服务适配。

必须明确：

- 输入字段（含必填 / 可选 / 校验规则）
- 输出字段
- 错误语义（错误码、错误类型、是否可重试）
- 幂等性
- 兼容性策略与版本策略
- 边界条件

### 1.6 错误处理

错误分层：

| 类型 | 含义 | 入口处理 |
|---|---|---|
| ValidationError | 参数 / 格式校验失败 | 400 |
| AuthError | 认证 / 授权失败 | 401 / 403 |
| DomainError | 业务规则违反 | 400 / 409（按子类） |
| InfrastructureError | DB / 外部 SDK / 网络错误 | 502 / 503 |
| SystemError | 未分类错误 | 500 |

要求：

- 错误在产生位置抛出对应类型
- Interface 层统一捕获并转换为响应
- 日志包含 trace id、关键参数（脱敏）
- 告警仅针对 SystemError 与高频 InfrastructureError

### 1.7 配置与环境

要求：

- 配置集中读取与校验
- 业务代码不读环境变量
- Secret 不进入代码、日志、测试快照
- 多环境差异通过环境变量切换，不通过代码分支
- 测试环境覆盖方式明确

### 1.8 测试架构

测试分层：

| 类型 | 覆盖 | mock 策略 |
|---|---|---|
| 单元测试 | Domain 规则、纯函数、关键 Application UseCase | Domain 不 mock；Application 中 mock Repository |
| 集成测试 | Repository、外部适配器、模块协作 | 用测试容器或测试数据库 |
| E2E 测试 | 关键用户路径 | 真实部署或 Docker Compose |

要求：

- Domain 必须有单元测试
- Repository 必须有集成测试
- 关键 API 路径必须有 E2E
- 架构调整必须先补测试再动结构

### 1.9 扩展点

仅为**已知真实变化点**保留扩展性：

| 候选 | 是否常见 | 提示 |
|---|---|---|
| 支付渠道 | 常见 | 用 PaymentProvider 接口 |
| 登录方式 | 常见 | 用 AuthStrategy 接口 |
| 存储后端 | 偶见 | 仅当真实迁移计划存在 |
| AI 模型供应商 | 常见 | 用 ModelProvider 接口 |
| 通知通道 | 常见 | 用 NotificationChannel 接口 |
| 多租户 | 视产品 | 初期不引入，除非首版即需要 |

**禁止**：因"未来可能需要"就抽象。规则：出现第二个真实用例时再抽象。

## 二、架构设计文档模板

写入位置：`docs/<branch>/design.md` 的「架构设计」节内部。

**边界说明**：

- 本模板**仅用于填充设计文档中的「架构设计」节内部内容**，不规定外层章节结构；外层结构以项目文档规则为准。
- 简单变更可只写其中部分子节（至少包含「模块设计」「分层与依赖」或一段方案概述），不必凑齐全部 20 节。
- 复杂变更应按下表完整填写。

```markdown
## 架构设计

### 1. 场景
0-1 初建 / 1-N 迭代（择一）

### 2. 目标
本次架构设计要解决什么。

### 3. 非目标
明确不做什么。

### 4. 不变量
哪些已有行为不能被破坏（1-N 必填；0-1 视情况）。

### 5. 约束
语言、框架、数据库、部署、合规、团队习惯。

### 6. 现状分析
1-N 必填；0-1 写「无历史系统」即可。
- 相关模块 / 调用链 / 数据流 / 测试覆盖

### 7. 核心模型
实体、关系、状态、规则、不变量。

### 8. 架构判断（1-N 专用）
不调整 / 局部调整 / 系统级调整 — 择一，并说明依据。

### 9. 候选方案
至少 2 个方案，列优缺点与适用规模。

### 10. 推荐方案与取舍
明确推荐与理由，并说明放弃哪些备选与原因。

### 11. 模块设计
| 模块 | 职责 | 不负责 | 对外契约 |
|---|---|---|---|
| ... | ... | ... | ... |

### 12. 分层与依赖
- 分层列表与每层职责
- 允许 / 禁止的依赖

### 13. 数据设计
- 数据模型 / 关键表 / 关键字段
- 事务边界与一致性

### 14. 接口契约
- API / 函数 / 事件 / 错误模型

### 15. 错误处理
- 错误类型与转换规则

### 16. 配置与环境
- 配置来源、Secret 管理、多环境

### 17. 测试策略
- 单元 / 集成 / E2E 范围
- 调整场景：先补测试再动结构的具体顺序

### 18. 迁移策略
1-N 调整场景必填；0-1 写「初始化步骤」即可。
- 阶段划分
- 每阶段产物
- 兼容策略
- 回滚策略

### 19. 风险与取舍
- 已知风险（概率 / 影响 / 缓解）
- 主动放弃的能力或质量维度

### 20. 验收标准
- 功能验收
- 架构边界验收（依赖方向、规则集中度）
- 测试覆盖验收
```

## 三、检查清单（汇总）

### 3.1 0-1 检查清单

详见 `workflow-greenfield.md`。要点：

- [ ] 目标 / 非目标 / 不变量 / 约束 已明确
- [ ] 核心业务模型已写出
- [ ] 至少 2 个候选方案 + 推荐与取舍
- [ ] 模块边界、分层、依赖规则已定义
- [ ] 数据访问、接口契约、错误模型已定义
- [ ] 配置 / 测试策略已定义
- [ ] 风险与验收标准已写明

### 3.2 1-N 调整判断清单

详见 `workflow-iteration.md`。要点：

- [ ] 已完成现状调查（模块 / 调用链 / 数据流 / 测试 / issues）
- [ ] 已明确变化点类型
- [ ] 已逐项回答 14 项判断题
- [ ] 已选择 A 不调整 / B 局部 / C 系统级（**唯一**）
- [ ] 兼容策略、迁移步骤、回滚策略已写明
- [ ] 测试先于结构调整的顺序已写明

## 四、推荐提示词模板

### 4.1 0-1 初建提示词

```text
请按 architecture-design 技能为这个新系统做架构设计。

背景：
[描述产品目标、用户、核心能力]

约束：
[语言、框架、数据库、部署、团队习惯]

要求按 workflow-greenfield.md 的 10 个步骤输出：
1. 系统类型判定
2. 核心业务模型
3. 至少 2 个候选架构方案
4. 推荐方案与取舍
5. 模块边界
6. 分层与依赖
7. 数据访问模式
8. 接口契约草案
9. 配置与环境
10. 测试策略

最终按 reference.md 的「架构设计文档模板」写入 design.md「架构设计」节。
不写代码。不过度设计。所有决策必须说明理由。
```

### 4.2 1-N 迭代提示词

```text
请按 architecture-design 技能评估这次需求是否需要架构调整。

需求：
[描述本次变化]

请按 workflow-iteration.md 的步骤输出：
1. 现状调查（先读相关代码与 issues）
2. 变化点定位
3. 14 项调整判断 + 结论（A / B / C 唯一选择）
4. 若 A：说明现有结构如何承载
5. 若 B / C：候选方案、推荐、兼容、迁移、回滚、测试

最终按 reference.md 的「架构设计文档模板」写入 design.md「架构设计」节。
- 不默认重构
- 不破坏公共接口（除非显式迁移）
- 不引入无需求驱动的抽象
- 测试先于结构调整
```

## 五、最终准则

- 0-1 架构设计的重点：**建立正确的边界**。
- 1-N 架构设计的重点：**判断是否需要改变边界**。
- 好的架构设计应让人能看懂边界、AI 能在边界内写代码、测试能验证行为、变更能小步迁移。
