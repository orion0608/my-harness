# 1-N 迭代架构设计工作流

适用于：在已有系统上**评估是否需要架构调整**并执行调整。

入口：由 `SKILL.md` 路由进入。

**重要前提**：迭代不等于必然调整架构。默认偏向"不调整"，必须有明确证据才能升级到调整。

---

## 触发场景

| 触发 | 是否一定要架构调整 |
|---|---|
| 新增小型功能 | 通常不需要 |
| 修复缺陷 | 通常不需要 |
| 性能优化 | 看瓶颈在结构还是局部实现 |
| 安全加固 | 看是否影响权限模型或数据流 |
| 模块重构 | 看是否影响公开契约或跨模块依赖 |
| 技术栈升级 | 多数情况需要架构评估（兼容、迁移路径） |
| 数据模型变化 | 通常需要 |
| 外部依赖变化 | 通常需要 |
| 已散落的业务规则继续扩散 | 通常需要局部调整 |

---

## 工作流步骤

### 步骤 1：现状调查（必做，不可跳过）

在提出任何调整建议之前，必须先把以下内容查清。

```markdown
## 现状调查

### 相关模块
本次变化涉及哪些目录 / 文件 / 模块？

### 当前调用链
请求 / 事件如何流动？经过哪几层？

### 当前数据流
数据在哪里产生 / 转换 / 写入 / 读取？

### 当前业务规则位置
规则集中在领域层，还是已散落到 Controller / SQL / 前端？

### 当前测试覆盖
哪些行为有单元 / 集成 / E2E 测试？覆盖空白点在哪里？

### 当前约束（不变量）
哪些 API、数据结构、字段、错误码必须保持兼容？
哪些行为在生产数据上已被依赖？

### 当前已知技术债
与本次变化相关的已知问题（如有 issues.md，先读）。
```

**禁止**：不读现有代码、不看测试、不查 issues 就提架构调整。

### 步骤 2：定位变化点

明确本次需求改变的是什么类型：

| 变化类型 | 典型架构影响 |
|---|---|
| 新增入口（API / CLI / 页面） | 通常局部，新增 Application UseCase |
| 新增业务规则 | 增到对应 Domain 模块 |
| 新增状态或生命周期 | 改状态机，可能影响存储 |
| 新增数据关系 | 改数据模型，可能影响 Repository 与查询 |
| 新增外部依赖 | 新增 Infrastructure 适配器 |
| 新增性能要求 | 看是否结构性瓶颈 |
| 新增权限维度 | 可能影响多个 Application UseCase |
| 新增部署形态 | 可能影响配置、Infrastructure |

### 步骤 3：调整判断（核心决策）

逐项回答下表，得到最终结论。

```markdown
## 架构调整判断清单

1. 现有模块是否能自然承载新需求？           [是 / 否]
2. 是否需要新增核心业务概念或实体？         [是 / 否]
3. 是否需要改变已有状态机或生命周期？       [是 / 否]
4. 是否需要改变数据库结构？                 [是 / 否]
5. 是否需要改变公共 API 或事件契约？         [是 / 否]
6. 是否需要引入新的外部依赖？               [是 / 否]
7. 是否存在跨模块调用扩散的风险？           [是 / 否]
8. 是否存在业务规则重复或散落的现象？       [是 / 否]
9. 是否存在事务边界或一致性变化？           [是 / 否]
10. 是否存在权限模型变化？                   [是 / 否]
11. 是否存在性能 / 可靠性 / 安全的新约束？   [是 / 否]
12. 是否能仅通过局部修改完成？              [是 / 否]
13. 如果不调整架构，下一次相同方向的迭代会明显更困难吗？  [是 / 否]
14. 如果调整架构，是否有足够测试保护？      [是 / 否]
```

#### 结论选择

| 输入特征 | 结论 |
|---|---|
| 1 / 12 为"是"，2~11 大多为"否" | **A. 不调整架构** |
| 1~2 个模块边界需要重划，不影响公共契约 | **B. 局部架构调整** |
| 涉及核心模型、公共接口、数据流或多个模块 | **C. 系统级架构调整** |

**只能选一种**。禁止"顺手再做一些大重构"。

---

## 分流 A：不调整架构

适用于：现有结构能承载，只是在内部添加 / 修改少量逻辑。

输出格式：

```markdown
## 架构判断：不调整

### 原因
- 现有模块边界可承载本次变化
- 依赖方向无需改变
- 数据模型无需改变
- 风险可由局部测试覆盖

### 实现约束
- 不新增跨模块依赖
- 不修改公共接口
- 不改变已有数据结构
- 新代码应放入：[模块名 / 文件位置]

### 测试要求
- 新增单元测试覆盖：[规则 / 函数]
- 新增 / 复用集成测试覆盖：[流程]
- 回归测试范围：[路径]
```

完成判断即可进入实现，不需要写迁移步骤。

---

## 分流 B：局部架构调整

适用于：只影响单个模块或少数相邻模块；不改变系统整体风格、部署形态；不破坏公共契约。

### 输出内容

```markdown
## 架构判断：局部调整

### 调整原因
现有结构在 [具体维度] 上不足以承载本次变化：
- [证据 1：现有代码片段或现状描述]
- [证据 2]

### 调整范围
- 涉及模块：[列表]
- 涉及文件：[列表]
- 涉及接口：[列表]
- 涉及数据结构：[列表]

### 调整前
[简要描述当前结构 / 调用链 / 数据流]

### 调整后
[简要描述目标结构 / 调用链 / 数据流]

### 兼容策略
- 公开 API：保持兼容 / 引入新版本（说明）
- 数据：迁移脚本 / 双写过渡 / 不需要
- 内部接口：允许变更，调用方一并更新

### 迁移步骤
1. 补充测试锁定旧行为
2. 引入新接口或新边界
3. 迁移调用方到新接口
4. 删除旧路径
5. 收紧依赖规则（如有）
6. 更新文档

### 回滚策略
- 触发条件：[例如 P0 缺陷、性能回退超过 X%]
- 回滚动作：[例如 revert PR、关闭新接口入口]
- 数据回滚：[如有]

### 验证方式
- 单元测试：[范围]
- 集成测试：[范围]
- 关键 E2E 路径回归：[路径]
```

### 小步迁移要点

- 先补测试再动结构。
- 不要一次性把所有调用方迁完，按子流程逐步迁移。
- 旧路径与新路径在过渡期共存，明确切换时间点。

---

## 分流 C：系统级架构调整

适用于：

- 多个模块边界需要重划
- 数据模型或核心流程变化
- 需要引入新的架构层、事件机制、权限模型或部署模型
- 当前架构已无法支撑主要演进方向
- 调整影响公共接口或持久化数据

### 输出内容

```markdown
## 架构判断：系统级调整

### 背景
现有架构在 [维度] 上无法继续支撑：
- [问题 1，附证据]
- [问题 2，附证据]

### 目标
调整后要解决：
- [结构性问题 1]
- [结构性问题 2]

### 非目标
本次不解决：
- [问题 X，原因]
- [问题 Y，原因]

### 影响范围
- 模块：[列表]
- 接口：[列表，标记是否破坏兼容]
- 数据：[列表，标记是否需要迁移]
- 测试：[需要补强的范围]
- 部署 / 运维：[变化]

### 目标架构
[新模块、分层、依赖方向、数据流的简要描述与图示]

### 迁移策略
分阶段，每阶段独立可验证：

1. 阶段 1：[目标 / 产物 / 验收]
2. 阶段 2：[目标 / 产物 / 验收]
3. 阶段 3：[目标 / 产物 / 验收]
...

每个阶段满足：
- 旧功能仍可用
- 可独立部署
- 可独立回滚

### 兼容策略
- 公共 API：[版本策略 / 双版本并存 / 弃用计划]
- 数据：[迁移脚本 / 双写 / Backfill / 校验]
- 事件契约：[版本策略]

### 回滚策略
- 每阶段独立回滚
- 数据迁移可回滚或可幂等重放
- 切换流量的阶段必须有金丝雀或灰度方案

### 风险与取舍
- [风险 1：发生概率 / 影响 / 缓解措施]
- [风险 2]

### 验证策略
- 每阶段验收标准
- 全局回归范围
- 性能 / 一致性 / 安全验证手段
```

### 强制要求

- **禁止一次性重写**。系统级调整必须分阶段。
- **每阶段必须独立可上线、可回滚**。
- **每阶段必须先补测试再动结构**。
- **数据迁移必须可重放或可回滚**。

## 1-N 检查清单

提出调整建议前自检：

- [ ] 已读取相关现有代码
- [ ] 已识别当前模块边界与调用链
- [ ] 已识别当前数据流与事务边界
- [ ] 已读取相关 issues / devlog
- [ ] 已识别当前测试覆盖
- [ ] 已明确本次变化点
- [ ] 已逐项回答 14 项调整判断清单
- [ ] 已选择 A / B / C 中**唯一一个**结论
- [ ] 若 A，已说明现有结构如何承载
- [ ] 若 B / C，已说明调整范围、兼容、迁移、回滚、测试
- [ ] 已识别风险与不变量
- [ ] 已定义完成标准

---

## 常见误判

| 误判 | 正确做法 |
|---|---|
| "顺便重构一下" | 拒绝；只做本次需求驱动的改动 |
| "未来可能微服务化，先拆模块" | 拒绝；除非有明确拆分计划与时间表 |
| "现有架构丑陋，先重写再加功能" | 拒绝；先用测试锁住旧行为，小步迁移 |
| "改动很大，所以一次性重写更快" | 几乎从不成立；系统级调整必须分阶段 |
| "我读了一部分代码就够了" | 不够；至少把相关调用链、数据流、测试看完 |
| "兼容策略 = 我尽量不破坏" | 必须显式写出兼容策略与迁移步骤 |
| "调整后再补测试" | 错；测试先于结构调整 |

## 写入

按 `reference.md` 中的「架构设计文档模板」写入 `docs/<branch>/design.md` 的「架构设计」节。

若是分流 C，建议在 `docs/<branch>/plan.md` 中拆出每个阶段的实现计划（与 Superpowers `writing-plans` 流程配合）。
