# test-report.md 编写规范

## 创建时机

测试执行完成后创建。可以分阶段更新（单元测试 → 集成测试 → 验收测试）。

## 文件位置

`docs/<branch-id>/test-report.md`

## 模板

```markdown
# 测试报告

> 分支：`<branch-name>`
> 最后更新：YYYY-MM-DD HH:MM

## 1. 测试概览

| 指标 | 数值 |
|------|------|
| 测试总数 |       |
| 通过     |       |
| 失败     |       |
| 跳过     |       |
| 覆盖率   |       |

## 2. 测试范围

<!-- 本次测试覆盖的功能模块和场景 -->

## 3. 失败用例

| 用例名称 | 失败原因 | 是否阻塞发布 | 处理方式 |
|---------|---------|:----------:|---------|
|         |         |            |         |

## 4. 跳过用例

| 用例名称 | 跳过原因 |
|---------|---------|
|         |         |

## 5. 已知问题

<!-- 本次测试中发现但不在本次修复范围内的问题 -->

## 6. 结论

<!-- 是否可以发布/合并：是 / 否（附原因） -->
```

## 内容要求

### 测试概览

- 数据必须来自**实际执行结果**，不能估算
- 覆盖率如无法计算，注明"未计算"并说明原因

### 测试范围

- 列出实际覆盖的模块和场景
- 如果有未覆盖的模块，说明为什么没测

### 失败用例

- 每个失败用例必须包含：名称、失败原因、是否阻塞发布、处理方式
- 处理方式：已修复并重新测试 / 已记录为已知问题 / 待修复
- 阻塞发布的用例必须加粗标注

### 跳过用例

- 必须说明跳过原因（环境限制、数据不足、优先级低等）
- 不能说"暂时跳过"——说明具体原因

### 已知问题

- 不属于本次功能引入、但被本次测试发现的问题
- 如果已记录到 issues.md，标注关联编号

### 结论

- 明确给出"可以发布/合并"或"不建议发布/合并"的结论
- 如不建议，必须说明阻塞项
