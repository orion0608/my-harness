# requirement.md 编写规范

## 创建时机

功能开发开始时，需求明确后立即创建。

## 文件位置

`docs/<branch-id>/requirement.md`

## 模板

```markdown
# 需求文档

> 分支：`<branch-name>`
> 创建时间：YYYY-MM-DD HH:MM

## 1. 用户原始需求

<!-- 记录用户的原话，不要改写 -->

## 2. 需求澄清

<!-- 经过对话澄清后的完整需求理解，包括：
  - 功能边界（做什么、不做什么）
  - 用户场景和使用流程
  - 性能/安全/兼容性等非功能需求
-->

## 3. 成功标准

<!-- 可验证的完成标准，尽量量化：
  - 功能层面：用户能完成什么操作
  - 质量层面：性能指标、测试覆盖率等
  - 验收条件：如何判断"做完了"
-->
```

## 内容要求

### 用户原始需求

- 使用用户原话，不添加自己的理解
- 如果用户分多条消息补充，逐条记录

### 需求澄清

- 必须是对话中**明确确认过**的内容，不可自行推测
- 明确标注功能边界：包含什么、不包含什么
- 如果涉及 UI，描述用户可见的交互流程

### 成功标准

- 每个标准必须是**可验证的**（能通过测试或演示确认）
- 避免模糊表述："好用"、"快速" 需量化
- 覆盖功能、质量、验收三个维度
