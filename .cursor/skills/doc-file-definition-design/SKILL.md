# design.md 编写规范

## 创建时机

头脑风暴完成、设计定稿后创建。

## 文件位置

`docs/<branch-id>/design.md`

## 内容要求

### 背景与目标

- 说清楚"为什么做"，不只是"做什么"
- 一句话讲清目标用户和核心价值

### 方案概述

- 高度概括，让新人快速理解整体思路
- 如果方案简单，2-3 句话即可

### 方案对比

- 至少列出 **被否决的方案**（为什么没选），不光是选中的
- 选择原因要具体：性能？维护性？团队熟悉度？

### 前端设计

- 涉及 UI 时**必填**；美学模式与实现方式须来自 AskQuestion 或用户明确指定
- 参考式须填 slug；不可因无 `package.json` 默认纯 HTML
- 与 `方案对比` 中的 UI/技术方案须一致

### 架构设计

- 弱约束：本节深度随变更范围而定，本技能不规定子节结构。
- 不涉及架构级变更时：1~3 段话覆盖"涉及的模块 / 关键数据流 / 调用链"即可。
- 涉及架构级变更时：按子节展开（场景、目标、不变量、模块设计、分层与依赖、接口契约、迁移策略、风险与验收等）。
- 涉及 UI 时：附上布局草图或组件树；具体交互与风格以「前端设计」表及方向预览/线框为准。

### 关键决策

- 记录有争议的决策及其理由
- 决策要包含上下文——未来的人需要知道"当时为什么这么选"

### 风险与约束

- 必须是当前已知的、具体的风险
- 不说"可能有问题"，说"如果 X 发生，会导致 Y"
