# release-notes.md 编写规范

## 创建时机

分支合并/发布前创建，作为 `CHANGELOG.md` 的草稿来源。

## 文件位置

`docs/<branch-id>/release-notes.md`

## 模板

```markdown
# 发布说明

> 分支：`<branch-name>`
> 版本：`vX.Y.Z`
> 发布日期：YYYY-MM-DD

## 1. 新增功能

- <!-- 功能 1 -->
- <!-- 功能 2 -->

## 2. 修复的问题

- <!-- 修复 1 -->
- <!-- 修复 2 -->

## 3. 破坏性变更

<!-- 如果有不兼容的变更，在此说明 -->

## 4. 已知限制

<!-- 本次发布中未解决但已知的限制 -->

## 5. 升级注意事项

<!-- 用户升级时需要了解的操作步骤或注意事项 -->

## 6. 依赖变更

| 依赖 | 旧版本 | 新版本 | 原因 |
|------|-------|-------|------|
|      |       |       |      |
```

## 内容要求

### 新增功能

- 面向**用户**描述，使用用户能理解的语言
- 按重要性排序，最重要的功能放前面
- 每个条目 1-2 行，不要技术实现细节

### 修复的问题

- 如果有关联的 issue 编号，标注 `（#123）`
- 描述修复后的用户可见效果，而非技术手段

### 破坏性变更

- 如果有不兼容的 API 变更、配置变更、行为变更，必须在此明确标注
- 说明旧行为、新行为、迁移方法

### 已知限制

- 诚实列出——比用户自己发现后再问要好
- 如果有计划在未来版本解决的，可以备注

### 升级注意事项

- 如果升级需要用户执行特定操作（如数据迁移、配置更新），写清步骤
- 如果是自动升级，注明"无需手动操作"

### 依赖变更

- 记录新增、移除、版本变更的依赖
- 注明变更原因
