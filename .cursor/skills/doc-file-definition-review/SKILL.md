# review.md 编写规范

## 创建时机

每次代码审查完成后追加一条记录。文件在首次审查完成时创建。

## 文件位置

`docs/<branch-id>/review.md`

## 模板

```markdown
# 审查记录

> 分支：`<branch-name>`

---

## YYYY-MM-DD HH:MM

**审查范围：**
<!-- commit 范围或文件范围 -->

**审查方式：**
<!-- 自审 / AI 审查 / 人工审查 -->

**发现问题：**

| # | 严重程度 | 问题描述 | 涉及文件 | 处理结果 |
|---|---------|---------|---------|---------|
| 1 | 严重     |         |         | 已修复   |
| 2 | 建议     |         |         | 已采纳   |
| 3 | 建议     |         |         | 不采纳   |

**不采纳原因（如有）：**
<!-- 对于标记为"不采纳"的建议，说明原因 -->

**总结：**
<!-- 本次审查的整体评价和改进方向 -->

---
```

## 内容要求

### 审查范围

- 精确到 commit hash 范围或文件路径
- 如果是分批审查的，标注这是第几批

### 审查方式

- 记录审查来源：自审、AI 审查、同事审查
- AI 审查可以标注使用的工具

### 发现问题

- 严重程度分为：**严重**（必须修复）、**建议**（推荐修复）、**可选**（可修可不修）
- 每个问题必须关联具体文件和行号
- 处理结果：已修复、已采纳、不采纳、待处理

### 不采纳原因

- 必须解释为什么不采纳
- 避免模糊理由，给出具体的技术或业务考量

### 总结

- 1-2 句话总体评价本次审查的代码质量
- 如果有反复出现的问题模式，在这里指出
