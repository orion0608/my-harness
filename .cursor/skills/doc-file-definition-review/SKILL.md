# review.md 编写规范

## 创建时机

每次代码审查完成后追加一条记录。文件在首次审查完成时创建。

## 文件位置

`docs/<branch-id>/review.md`

## 模板

```markdown
# 审查记录

> 分支：`<branch-name>`

---

## YYYY-MM-DD HH:MM

**审查范围：**
<!-- commit 范围或文件范围 -->

**审查方式：**
<!-- 自审 / AI 审查 / 人工审查 -->

**前端验证：**（涉及 Web/UI 时必填；无则写「不适用」）
- 验证层级：P（呈现） / I（联动）
- 判定依据：
- 环境：（服务与 URL；I 级须为真实应用入口）
- 覆盖路径：（逐条；I 级注明数据已加载或写操作是否成功）
- 浏览器证据：（snapshot / screenshot 结论；降级须说明原因）
- 控制台/网络：（browser_cdp 采集摘要；error 须列出）
- 未覆盖：（若仅 P 或联动阻塞）

**发现问题：**

| # | 严重程度 | 问题描述 | 涉及文件 | 处理结果 |
|---|---------|---------|---------|---------|
| 1 | 严重     |         |         | 已修复   |
| 2 | 建议     |         |         | 已采纳   |
| 3 | 建议     |         |         | 不采纳   |

**不采纳原因（如有）：**
<!-- 对于标记为"不采纳"的建议，说明原因 -->

**总结：**
<!-- 本次审查的整体评价和改进方向 -->

---
```

## 内容要求

### 审查范围

- 精确到 commit hash 范围或文件路径
- 如果是分批审查的，标注这是第几批

### 审查方式

- 记录审查来源：自审、AI 审查、同事审查
- AI 审查可以标注使用的工具

### 前端验证

- 按 **`review-frontend-enhancement`** 填写；I 级须证明联动路径（非仅静态 Demo）
- 「浏览器证据」须对应实际调用的 `cursor-ide-browser` 步骤摘要，不可臆造
- 「控制台/网络」须来自 `browser_cdp` 采集结果（见 **`review-frontend-enhancement`**）

### 发现问题

- 严重程度分为：**严重**（必须修复）、**建议**（推荐修复）、**可选**（可修可不修）
- 每个问题必须关联具体文件和行号
- 处理结果：已修复、已采纳、不采纳、待处理

### 不采纳原因

- 必须解释为什么不采纳
- 避免模糊理由，给出具体的技术或业务考量

### 总结

- 1-2 句话总体评价本次审查的代码质量
- 如果有反复出现的问题模式，在这里指出
